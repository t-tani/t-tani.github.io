C++のクラスとリバースエンジニアリング

C++を逆アセンブルした際には、アセンブラコードにはメンバー関数であることやクラスの宣言部であることは明示されていない。メモリの使用量や関数の構造から、クラスであることを判断する必要がある。簡単な C++のコードをコンパイル/逆アセンブルして this ポインターやクラスのメンバー関数の呼び出し、メンバー変数の呼ばれ方を確認してみる。なお、MSVS2017 と GCC でコンパイルした PE ファイルを利用する。

クラスを含むソース（sample）

  • sample_class.cpp
#include "windows.h"
#include "tchar.h"
#include <stdio.h>

class Employee
{
  public:
    int number;
    char name[128];
    long pay;
    void ShowData();
    void Test();
};

void Employee::ShowData()
{
    printf("number: %d\n", number);
    printf("name: %s\n", name);
    printf("pay: %d\n", pay);
    Test();
}

void Employee::Test()
{
    printf("Test function\n");
    return;
}

//Employee t_tani;
int main(int argc, char *argv[])
{
    Employee t_tani;

    printf("size: %X\n", sizeof(Employee));
    t_tani.number = 0x1111;
    _tcscpy(t_tani.name, _T("t_tani"));
    t_tani.pay = 0x100;
    t_tani.ShowData();

    return 0;
}

this ポインターの分析

リバースエンジニアリングする上で構造体とクラスが持つ共通として、メモリ上でスタックサイズを増やして変数を使用する共通点がある。main() の以下のコードで 0x94 を積んでいるので、この関数では 0x94 バイトのデータの塊を使用することが分かり、構造体かクラスで使われることが推測される。

  • スタックを積むコード
55                        push    ebp
8B EC                     mov     ebp, esp
81 EC 94 00 00 00         sub     esp, 94h

次に var_90 に着目する。以下のコードを見ると、var_90 に直接 0x1111 を代入している。これは、最初のメンバー変数 t_tani.number = 0x1111; に該当する部分である。そして、その後に var_94 のアドレスを ECX に入れて関数に呼び出し call を行っている。通常、クラスのメンバー関数を呼び出すときには、現在のメモリの塊のポインターを ECX にいれて呼び出す(呼び出し規約:__thiscall)。そのため、 ECX に入った var_90 が C++での this ポインターになる。

  • main()でメンバー変数を使用する部分
C7 85 70 FF FF FF 11 11+  mov     [ebp+var_90], 1111h
(snip)
8D 8D 70 FF FF FF         lea     ecx, [ebp+var_90]
E8 CD FB FF FF            call    sub_401145

次にメンバー関数 ShowData()を確認してみる。ECXに this ポインターが渡されたことを考慮して確認すると ECX に入ったポインターが var_4 変数に入り、この変数のポインターを増加させながら、クラスの変数を使用することになる。

  • クラスの 1 番目のメンバー変数を使用部分
89 4D FC              mov     [ebp+var_4], ecx
8B 45 FC              mov     eax, [ebp+var_4]
8B 08                 mov     ecx, [eax]
51                    push    ecx
68 30 6B 40 00        push    offset aNumberD ; "number: %d\n"
E8 A4 FB FF FF        call    sub_401032

はじめのメンバー変数は int だったため、var_4 を EDX に入れて、4 バイト加算し、次の変数を指すようにしている。

  • クラスの 2 番目のメンバー変数を使用部分
8B 55 FC              mov     edx, [ebp+var_4]
83 C2 04              add     edx, 4
52                    push    edx
68 3C 6B 40 00        push    offset aNameS   ; "name: %s\n"

次に 84h バイト加算して、変数を指している。これは 2 番目の変数が char name[128] であり、1 つ目のメンバー変数と 2 つ目のメンバー変数を加算(4h+80h)したものを足している。なお、80h は 10 進数で 128 である。さらに、ECXに this ポインターである var_4 を渡して、関数を呼んでいる。この部分は Employee::Test() を呼び出す部分である。

コンストラクターとデストラクター

コンストラクターを探すときのコツとして、コンストラクターの使われ方をもとに探すと良い。コンスタクターは主に、クラスで用いる変数の初期化で使われることが多く、複数の変数をまとめて初期化するような処理があるものもあり、その周辺に memset() や malloc() などが見えた場合にはコンストラクターである可能性が非常に高い。
デストラクターは通常コードの最後に位置するものであり、呼び出し前に ECX をクラスのオブジェクトポインターへ入れる。関数の中では、Close 関連の呼び出されることが多いため、そのように判断すると良い。

カプセル化

カプセル化された関数を呼ぶ際に、ECXにオブジェクトポインターを入れてから関数を呼ぶ点は他のメンバー関数の呼び出しと同じだが、一般のメンバー関数であるか、private 関数であるかの判別する方法はないため、区別することはほとんど不可能。

ポリモフィズム

ポリモフィズムのバイナリには特徴がある。仮想関数は一般的な関数が作成される方法とは異なり、別途テーブルが作成される。このテーブルの情報はコンストラクターとデストラクターでも管理される。このテーブルは .rdata セクションに保存される。
